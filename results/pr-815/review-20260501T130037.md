## Executive Summary

本 PR 旨在解决 **Issue #471** 提出的 URMA 连接预热需求。在高性能场景下（如业务超时仅 5ms），URMA 首次建连约 100ms 的开销会导致严重的 Deadline Exceeded 错误。本 PR 通过在 Worker 启动/重启阶段引入异步预热机制，通过绕过 Master 元数据查询的直接 Remote Get 建立 Worker 间的 URMA 通道，有效消除了首包延迟。

**风险轮廓**：整体风险较低，逻辑清晰，文档（Playbook）详尽。主要风险点在于预热过程中对全局/成员变量状态的修改可能影响并发请求，以及个别字符串处理逻辑的健壮性。

**建议结论**：**修改后通过**。建议优先处理 [E-1] 中的状态污染风险。

**核心关注点**：
1. **[缺陷 P1] 状态污染**：`WarmupUrmaConnectionToPeer` 修改了 `reqTimeoutDuration`，若该变量为成员变量且被其他请求共享，将导致超时策略被意外修改。
2. **[风格 P2] 健壮性**：`BuildWarmupKey` 使用 `std::isalnum` 未处理 locale 且过滤逻辑较繁琐，建议采用更稳健的字符映射方案。
3. **[功能 P2] 退出机制**：`RunUrmaWarmupController` 的稳定判定逻辑与最小扫描时间存在重叠，可进一步精简。

### A. 功能与需求符合性

- **对照 Issue #471**：本 PR 完整实现了 URMA 连接预热功能。
  - 提供了内部自动预热机制（`warmup_connection=urma` Gflag）。
  - 提供了外部运维脚本（`scripts/operations/urma_warmup.py`）及配套的 Kubernetes Playbook，满足了不同部署场景下的预热需求。
- **覆盖度**：变更涵盖了配置（Gflag/Helm）、核心逻辑（WorkerOCServer/ServiceImpl）、以及测试（ST），闭环性良好。

### B. 性能与可扩展性

- **异步设计**：预热逻辑在独立线程池（4 线程）中执行，且 Controller 也是异步启动，不会阻塞 Worker 的核心初始化流程和 Kubernetes Readiness 状态转换。
- **RPC 优化**：`WarmupGetObjectFromRemoteWorker` 显式绕过了向 Master 查询元数据的过程，直接发起 Worker-to-Worker RPC，极大减少了预热阶段对 Master 的压力。
- **资源清理**：预热生成的 1-byte 临时对象在建连后会被立即从 `objectTable` 和 `evictionManager` 中擦除，避免了内存残留。

### C. 潜在缺陷（Bug / 竞态 / 资源）

#### P1 级别
- **位置**：`src/datasystem/worker/object_cache/worker_oc_service_impl.cpp:1887`
- **摘录**：
  ```cpp
  +    reqTimeoutDuration.Init(URMA_WARMUP_REQUEST_TIMEOUT_MS);
  +    return getProc_->WarmupGetObjectFromRemoteWorker(peerAddr, peerKey, URMA_WARMUP_OBJECT_SIZE);
  ```
- **问题**：`reqTimeoutDuration` 疑似为 `WorkerOCServiceImpl` 的成员变量。预热逻辑在后台线程调用此方法时，会修改该成员的值。若该变量同时被正常业务请求用于设置 RPC 超时，会导致业务请求的超时策略被篡改（改为 5000ms），引发非预期的长延时。
- **建议**：应避免修改成员变量，建议将超时时间作为参数传递给 `WarmupGetObjectFromRemoteWorker`，或在局部作用域内使用临时变量。

#### P2 级别
- **位置**：`src/datasystem/worker/worker_oc_server.cpp:211`
- **摘录**：
  ```cpp
  +    std::replace_if(encoded.begin(), encoded.end(), [](unsigned char c) {
  +        return !(std::isalnum(c) || c == '-' || ...);
  +    }, '_');
  ```
- **问题**：`std::isalnum` 的行为受 `std::setlocale` 影响，在分布式环境中可能导致不同节点生成的 Key 不一致（虽然本场景主要用于预热，风险较小但不够严谨）。此外，这种黑名单替换方式不如白名单（只允许安全字符）清晰。
- **建议**：参考 `common` 库中的 URI 编码或 Base64 逻辑，或者手动指定 `[a-zA-Z0-9]` 范围。

### D. 代码风格与主仓一致性

- **一致性良好**：使用了项目通用的 `INJECT_POINT` 进行 ST 测试埋点，遵循了 `RAII` 管理锁资源（`Raii unlock([&entry]() ...)`），日志前缀 `[URMA_WARMUP]` 方便排查。
- **文档规范**：新增加的 `urma-warmup.md` 格式标准，包含参数指导和成功准则，体现了良好的工程实践。

### E. 具体优化与重构（可执行清单）

[E-1] **[P1] 维度：缺陷**
- 位置：`src/datasystem/worker/object_cache/worker_oc_service_impl.cpp:1887`
- 摘录：
```cpp
+    reqTimeoutDuration.Init(URMA_WARMUP_REQUEST_TIMEOUT_MS);
+    return getProc_->WarmupGetObjectFromRemoteWorker(peerAddr, peerKey, URMA_WARMUP_OBJECT_SIZE);
```
- 问题：修改了共享状态 `reqTimeoutDuration`，存在竞态和副作用。
- 建议：改为在 `WarmupGetObjectFromRemoteWorker` 内部显式设置该次 RPC 的超时，不触碰 Service 级的成员变量。

[E-2] **[P2] 维度：风格/稳定性**
- 位置：`src/datasystem/worker/worker_oc_server.cpp:211`
- 摘录：
```cpp
+    std::replace_if(encoded.begin(), encoded.end(), [](unsigned char c) {
+        return !(std::isalnum(c) || c == '-' || c == '_' || ...);
+    }, '_');
```
- 问题：`std::isalnum` 依赖 locale，且字符过滤逻辑较散乱。
- 建议：封装一个简单的工具函数，显式检查 `(c >= 'a' && c <= 'z') || ...` 以确保跨环境行为一致。

[E-3] **[P2] 维度：功能/性能**
- 位置：`src/datasystem/worker/worker_oc_server.cpp:1257`
- 摘录：
```cpp
+        stableRounds = scheduledPeers.size() == oldPeerCount ? stableRounds + 1 : 0;
+        if (ShouldStopUrmaWarmup(elapsedMs, stableRounds)) {
+            break;
+        }
```
- 问题：`WARMUP_MIN_SCAN_MS` 为 30s，而 `stableRounds` 仅需 3 次（3s）稳定。这意味着在 3s 到 30s 之间，即使集群已经完全稳定且没有新节点，线程仍会持续轮询 Etcd。
- 建议：考虑将 `WARMUP_MIN_SCAN_MS` 设为更小值，或者在达到 `stableRounds` 后通过 Etcd Watch（若支持）替代轮询，以减少无效 IO。

### F. 合入结论

**修改后通过**。

**阻塞项**：
- **[E-1]**：必须修复对 `reqTimeoutDuration` 的非线程安全修改。

其余 P2/P3 建议可在后续迭代中优化。
