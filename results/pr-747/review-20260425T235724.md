### Executive Summary

本 PR 为 **yuanrong-datasystem** 引入了 URMA 性能日志采样功能（`urma_enable_perf`），并增强了分布式 RDMA 操作的延迟观测能力。
**风险轮廓**：
1. **P0 级阻塞项**：`worker_worker_oc_service_impl.cpp` 中引用了未定义的枚举值 `WORKER_RPC_GET_REMOTE_OBJECT_LATENCY`，将导致编译失败。
2. **P1 级兼容性风险**：`kv_metrics.cpp` 中采用插入式更新导致后续所有指标 ID 偏移。若监控系统或共享内存指标 Buffer 依赖原始 ID，将造成严重的观测数据错乱。
3. **性能与配置**：`err_timeout` 从典型值硬编码为 8 (128ms)，在网络波动场景下可能引入过早的超时报错，需确认业务容忍度。

建议：**不通过**。须先修复编译错误并调整指标 ID 分配策略。

---

### A. 功能与需求符合性

- **URMA 性能采样**：通过 `urma_enable_perf` 开关周期性调用 `urma_get_perf_info` 并输出至日志。满足对底层 RDMA 库运行状态的深度观测需求。
- **延迟观测增强**：新增了 `WORKER_URMA_WAIT_LATENCY` 指标，覆盖了 `WaitToFinish` 的等待开销，是对原有 `WORKER_URMA_WRITE_LATENCY` 的重要补充。
- **缺陷/缺口**：
  - 指标定义缺失：在 `worker_worker_oc_service_impl.cpp` 增加了埋点，但未在 `kv_metrics.h` 声明对应枚举。
  - 配置一致性：`worker_config.json` 与 Helm `values.yaml` 均同步增加了配置项，符合标准发布规范。

### B. 性能与可扩展性

- **采样线程开销**：`PerfThreadMain` 每 10s 运行一次，且采样过程通过专用子线程异步处理，对 Worker 主逻辑热路径无性能影响。
- **日志压力**：`LOG(INFO)` 输出的性能信息（预设 16KB 缓冲区）在开启采样时可能导致日志量激增。建议在 `PerfThreadMain` 中对返回的消息长度做二次校验，防止异常超长字符串污染日志。
- **锁粒度**：`perfThread_` 的启动与停止逻辑位于 `Init` / `Stop` 阶段，不涉及高频竞争锁，可扩展性良好。

### C. 潜在缺陷（Bug / 竞态 / 资源）

#### P0 级问题
- 位置：`src/datasystem/worker/object_cache/worker_worker_oc_service_impl.cpp:660`
- 摘录：
```cpp
+    METRIC_TIMER(metrics::KvMetricId::WORKER_RPC_GET_REMOTE_OBJECT_LATENCY);
```
- 问题：`WORKER_RPC_GET_REMOTE_OBJECT_LATENCY` 在 `src/datasystem/common/metrics/kv_metrics.h` 中**未定义**，会导致构建中断。
- 建议：在 `KvMetricId` 枚举中添加该定义，并在 `kv_metrics.cpp` 的 `KV_METRIC_DESCS` 数组中增加对应描述。

#### P1 级问题
- 位置：`src/datasystem/common/metrics/kv_metrics.cpp:44`
- 摘录：
```cpp
     { 17, "worker_urma_write_latency", MetricType::HISTOGRAM, "us" },
+    { 18, "worker_urma_wait_latency", MetricType::HISTOGRAM, "us" },
-    { 18, "worker_tcp_write_latency", MetricType::HISTOGRAM, "us" },
+    { 19, "worker_tcp_write_latency", MetricType::HISTOGRAM, "us" },
```
- 问题：**破坏了指标 ID 的向后兼容性**。该项目若存在依赖固定 ID 的监控看板（如 Grafana 模板、外部采集插件或共享内存二进制导出格式），此变更会导致 18 号之后的所有监控数据语义完全错乱。
- 建议：将新指标 `WORKER_URMA_WAIT_LATENCY` 放置在 `KvMetricId` 枚举的最末尾（`KV_METRIC_END` 之前），分配当前最大可用 ID。

#### P2 级问题
- 位置：`src/datasystem/common/rdma/urma_resource.cpp:215`
- 摘录：
```cpp
-    jfsConfig.err_timeout = URMA_TYPICAL_ERR_TIMEOUT;
+    constexpr uint8_t errTimeout = 8;  // this timeout level means 128ms
+    jfsConfig.err_timeout = errTimeout;
```
- 问题：弃用通用典型值 `URMA_TYPICAL_ERR_TIMEOUT` 改为硬编码 128ms。在 RDMA 网络拥塞或大规模重传时，较短的硬件超时（Retry Timeout）可能导致连接过早进入 Error 状态，建议说明改动背景或保留配置能力。

### D. 代码风格与主仓一致性

- **日志阈值**：`zmq_service.cpp` 中将拥塞警告阈值硬编码为 5ms（`LOG_WARNING_THRESHOLD`），与原有的 `RPC_POLL_TIME` 逻辑不一致。建议将此阈值通过 `FLAGS` 或配置下发，避免在不同硬件环境下产生过多无效告警日志。
- **变量命名**：`PerfThreadMain` 中的局部变量 `len` 初始化为 `perfBufferLen`，调用 `ds_urma_get_perf_info` 后其值可能被底层库修改，符合 C 风格 API 惯例。

```cpp

+        std::vector<char> buffer(len);
```
