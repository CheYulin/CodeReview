## Executive Summary

本 PR 主要进行了 **构建系统重构与依赖标准化**，重点解决了 `jemalloc` 在共享库与 Python 扩展中的链接复杂性。通过移除自定义的 `jemalloc_dynamic` 转换逻辑，并统一利用 `cc_shared_library` (`//:datasystem`) 导出符号，显著简化了构建链条。

**风险轮廓**：低至中。主要风险在于 `export_datasystem_symbols.lds` 中过度放开的 RTTI/VTable 符号导出可能导致共享库体积膨胀或符号冲突，但对于解决跨共享库边界的 C++ 异常与类型识别是必要的。

**核心建议**：
1. **P1 级**：核实 `_ZTI*` 和 `_ZTV*` 的全局导出是否过于宽泛，建议限制在 `datasystem::` 命名空间内。
2. **P2 级**：`third_party/jemalloc.BUILD` 的重构简化了流程，但需确认 `--disable-initial-exec-tls` 的移除（在 static 编译阶段）是否会对某些旧版动态加载场景产生性能影响。
3. **P2 级**：Python 扩展现在动态依赖 `//:datasystem`，需确保安装包中正确包含该 .so 及其运行时路径（RPATH）。

---

## A. 功能与需求符合性

本 PR 未关联特定 Issue/RFC，但从代码变更推断，其目标是 **“解决共享库链接缺失与 jemalloc 符号一致性问题”**。
- **覆盖度**：成功移除了复杂的 `bazel/build_python.bzl` 宏，通过 `dynamic_deps` 实现了更符合 Bazel 现代标准的共享库依赖模式。
- **缺口**：在 `src/datasystem/worker/BUILD.bazel` 中，`add_miss_libs_fixme` 依然存在且由于修复链接问题而膨胀，这表明底层依赖关系仍有待进一步解耦，本 PR 仅是“止痛药”。

---

## B. 性能与可扩展性

- **Jemalloc 链接**：将 `jemalloc` 静态链接进 `datasystem` 共享库并统一管理，避免了 Python 进程中出现多份 malloc 实现导致的内存统计混乱或竞态，对分布式系统长时运行的稳定性有正面作用。
- **符号导出**：`_ZTI*` (RTTI) 和 `_ZTV*` (VTable) 的显式导出虽然增加了 `datasystem.so` 的符号表大小，但解决了多线程/多插件场景下 `dynamic_cast` 失败的潜在风险。

## C. 潜在缺陷（Bug / 竞态 / 资源）

### P1 级

#### 1. 符号导出范围过广
- **位置**：`export_datasystem_symbols.lds:8`
- **摘录**：
```lds
+    _ZTI*;        # 显式导出所有 RTTI 符号
+    _ZTV*;        # 显式导出所有虚表符号（如果 Buffer 有虚函数）
```
- **问题**：在 `global:` 作用域下使用通配符 `_ZTI*` 会导出链接器可见的所有第三方库的 RTTI 符号。这可能导致与宿主进程或其他插件的符号冲突。
- **建议**：建议限定前缀或将其移入 `extern "C++"` 块内，例如使用 `_ZTIN10datasystem*` (对应 `datasystem::` 命名空间) 以缩小范围。

---

## D. 代码风格与主仓一致性

- **Bazel 规范**：移除了自定义 rule `_jemalloc_pybind_extension`，转而使用官方推荐的 `dynamic_deps`，提升了与主仓及下游项目的兼容性。
- **LDS 文件**：`export_datasystem_symbols.lds` 中引入了注释，与项目中现有的简洁 LDS 风格略有差异，但有助于维护。
