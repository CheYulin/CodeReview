### Executive Summary

本 PR 为 **openeuler/yuanrong-datasystem** 引入了针对 URMA（快速传输）失败回退到 TCP 路径时的流量限制机制（`UrmaFallbackTcpLimiter`）。主要风险点在于客户端侧的状态跟踪 Map 存在潜在的内存泄漏风险，且限流阈值（10MB/1MB）为硬编码，在高性能场景下可能过于保守。建议在修复状态清理逻辑并增加配置项后合入。

1. **修复内存泄漏**：`ClientWorkerRemoteApi` 中的 `urmaFallbackStatus_` 在某些异常路径下无法清理，需确保所有请求结束路径均调用 `Consume` 或 `Clear`。
2. **限流参数可配置**：将 10MB 的总限制和 1MB 的单请求限制改为配置项。
3. **优化 Key 类型**：避免在 Map 中使用 `ObjectBufferInfo` 的裸指针作为 Key。

---

### A. 功能与需求符合性

本变更满足了分布式缓存系统在高性能网络抖动或配置错误时，保护共享 RPC 通道的需求。通过 RAII 机制（`Ticket`）在 `ShmGuard` 和 RPC 生命周期内跟踪回退字节数，实现了细粒度的反压。
- **覆盖性**：涵盖了 Client-to-Worker, Worker-to-Client, Worker-to-Worker 三类回退场景。
- **缺口**：对于单次 1MB 以上的对象，URMA 失败将直接导致请求失败（被限流器拒绝），需确认这是否符合业务对“最大努力回退”的预期。

---

### B. 性能与可扩展性

- **性能**：采用了原子变量 `atomic<uint64_t>` 和 RAII 票据，在数据路径上的开销极低。
- **可扩展性**：`UrmaFallbackTcpLimiter` 设计为通用组件，但硬编码的 10MB 限制在大规模集群或 100G 网络环境下可能成为瓶颈。

### C. 潜在缺陷（Bug / 竞态 / 资源）

#### P1 级缺陷
- **位置**：`src/datasystem/client/object_cache/client_worker_api/client_worker_remote_api.cpp`
- **摘录**：
  ```cpp
  +    std::unordered_map<const ObjectBufferInfo *, Status> urmaFallbackStatus_;
  ```
- **问题**：**内存泄漏风险**。`SendBufferViaUb` 失败时会将状态存入该 Map，但如果 `PreparePublishReq` 后续逻辑报错导致 `Publish` 提前返回，`ConsumeUrmaFallbackStatus` 将永远不会被调用，导致 Map 条目残留。
- **建议**：在 `ObjectBufferInfo` 生命周期结束时确保清理，或将状态直接存储在 `ObjectBufferInfo` 内部（如果允许修改其定义）。

#### P1 级缺陷
- **位置**：`src/datasystem/common/object_cache/urma_fallback_tcp_limiter.h`
- **摘录**：
  ```cpp
  +    static constexpr uint64_t kMaxPendingBytes = 10 * 1024 * 1024;
  +    static constexpr uint64_t kMaxSinglePayloadBytes = 1024 * 1024;
  ```
- **问题**：**硬编码限制**。10MB 的全局回退限制对于高吞吐 Worker 进程过小，且 1MB 的单请求限制可能导致正常大小的对象在 URMA 偶发故障时无法回退。
- **建议**：改为 `GFLAGS` 或从配置文件读取，允许根据机型和网络带宽调整。

#### P2 级缺陷
- **位置**：`src/datasystem/client/object_cache/client_worker_api/client_worker_remote_api.h`
- **问题**：使用 `const ObjectBufferInfo *` 裸指针作为 Map 的 Key 是不安全的。虽然目前生命周期基本覆盖，但若底层存在对象池化或内存复用，可能导致状态错乱。

[E-1] [P1] 维度：缺陷
- 位置：`src/datasystem/client/object_cache/client_worker_api/client_worker_remote_api.cpp:436` 附近

- 建议：添加 `DEFER` 逻辑或在 `Publish` 退出前强制调用 `Consume/Clear`。
