## Executive Summary

本 PR 为 ZMQ RPC 框架引入了轻量级的度量（Metrics）系统，旨在解决 **Issue #367** 提出的增强可观测性需求。

**风险轮廓**：低风险。变更主要集中在非阻塞的统计指标收集和日志增强，对核心逻辑无破坏性修改。
**核心贡献**：
1.  **细粒度观测**：实现了序列化/反序列化时延、I/O 系统调用时延的直方图统计（Histogram），以及套接字事件和错误分类计数（Counter）。
2.  **错误分类**：引入了 `IsNetworkErrno` 辅助函数，能够区分网络级硬错误与 `EAGAIN` 等预期内状态，提升了故障定位效率。
3.  **完备测试**：提供了覆盖 20 个测试用例的单元测试，验证了并发安全性及各类故障场景下的指标正确性。

**阻塞项**：无 P0/P1 级阻塞项。建议针对 `EAGAIN` 统计逻辑的严密性（E-1）进行微调。

---

### A. 功能与需求符合性

-   **需求对齐**：本 PR 直接实现了 **Issue #367** 要求的可观测性框架。通过在 `ZmqSocketRef` 和 `ZmqMonitor` 中埋点，补齐了原框架在 RPC 层级缺乏性能指标和网络异常分类统计的短板。
-   **覆盖完整性**：涵盖了发送/接收、成功/失败、时延/计数三个维度的监控，与分布式系统对 RPC 层的监控预期高度吻合。

### B. 性能与可扩展性

-   **开销控制**：在 `zmq_metrics_def.h` 中通过注释声明了时延直方图约 70ns 的开销，这在微秒级的 RPC 路径上是可以接受的。
-   **无锁设计**：指标收集依赖于 `common/metrics` 底层实现（通常为原子操作或 ThreadLocal），在大规模并发下扩展性良好。
-   **扩展性**：`ZMQ_METRIC_DESCS` 数组化定义方便后续增加新的观测点。

### C. 潜在缺陷（Bug / 竞态 / 资源）

#### P2 级问题
-   **指标漏计风险**：在 `zmq_socket_ref.cpp` 中，`ZMQ_M_RECV_EAGAIN` 的计数逻辑依赖于 `flags == ZmqRecvFlags::NONE`。若用户显式传递了其他标志（如某种自定义封装标志），则即便发生 `EAGAIN` 也不会计数。

#### P3 级问题
-   **日志类型匹配**：`zmq_socket.cpp` 中将 `int64_t` 的 `waitSec` 传入 `%d` 格式字符串，可能在特定编译器下触发警告。

### D. 代码风格与主仓一致性

-   **命名一致性**：遵循了 `ZMQ_M_` 前缀的枚举命名规范。
-   **依赖管理**：在 `BUILD.bazel` 中正确隔离了 `zmq_metrics_def` 库，避免了头文件循环依赖。
-   **规范引用**：使用了标准的 Huawei Open Source 版权头。

### E. 具体优化与重构（可执行清单）

**[E-1] [P2] 维度：功能/缺陷**
- 位置：`src/datasystem/common/rpc/zmq/zmq_socket_ref.cpp`:149
- 摘录：
```cpp
+        if (e == EAGAIN) {
+            if (flags == ZmqRecvFlags::NONE) {
+                metrics::GetCounter(ZMQ_M_RECV_EAGAIN).Inc();
+            }
+        }
```
- 问题：`ZMQ_M_RECV_EAGAIN` 的定义（见 `zmq_metrics_def.h`）是为了捕捉阻塞模式下的超时。但 ZMQ 的 `EAGAIN` 也常出现在非阻塞调用中。当前检查 `flags == ZmqRecvFlags::NONE` 过于苛刻，且未考虑未来标志位的扩展。
- 建议：改为判断 `e == EAGAIN` 即可。如果仅关注阻塞模式超时，应检查套接字本身是否处于阻塞属性，而非仅仅看单次调用的 `flags`。

**[E-2] [P3] 维度：风格/一致性**
- 位置：`src/datasystem/common/rpc/zmq/zmq_socket.cpp`:78
- 摘录：
```cpp
+        int64_t waitSec = std::chrono::duration_cast<std::chrono::seconds>(endTick - startTick).count();
+        LOG(WARNING) << FormatString("[ZMQ_RECV_TIMEOUT] Blocking recv timed out after %d seconds", waitSec);
```
- 问题：`int64_t` 与 `%d` 不匹配。
- 建议：显式转型为 `int`（通常秒数不会溢出）或使用更合适的格式占位符：
  `FormatString("[ZMQ_RECV_TIMEOUT] ... after %ld seconds", static_cast<long>(waitSec))`。

**[E-3] [P2] 维度：功能**
- 位置：`src/datasystem/common/rpc/zmq/zmq_metrics_def.h`:59
- 摘录：
```cpp
+inline bool IsNetworkErrno(int e)
+{
+    return e == ECONNREFUSED || e == ECONNRESET || e == ECONNABORTED ||
+           e == EHOSTUNREACH || e == ENETUNREACH || e == ENETDOWN ||
+           e == ETIMEDOUT || e == EPIPE || e == ENOTCONN;
+}
```
- 问题：该列表覆盖了常见的 TCP 错误，但对于 ZMQ 专有的 `EFSM`（状态机错误）或某些底层 IPC 错误未作处理。虽然当前列表足够应对 90% 场景，但建议在文档中说明这是针对“网络传输层”的过滤。
- 建议：保持现状，但在 `ZMQ_M_NET_ERROR` 注释中明确其针对的是上述定义的典型网络错误。
