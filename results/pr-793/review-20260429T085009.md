### Executive Summary

本 PR 旨在增强 ZMQ RPC 的全链路延迟观测能力，引入了 7 个关键阶段的耗时统计（从客户端入队到服务端响应发送）。
**核心改进**：
1. **性能优化**：将客户端响应处理中的指标计算从多次 `FindTickTs` 遍历（$O(k \cdot N)$）优化为单次 $O(N)$ 遍历。
2. **异步上下文保留**：通过在 `AsyncWrite` 时快照 `MetaPb` 并在 `AsyncRead` 时合并，解决了异步调用下客户端原始 Tick 丢失导致无法计算 E2E 延迟的问题。
3. **测试覆盖**：提供了完整的 ST 测试和 REPL 工具，验证了延迟分解的数学一致性。

**风险预警**：
1. **P1 级语义退化**：`TICK_CLIENT_SEND` 被从 Frontend 线程移至用户线程 `AsyncWrite` 处，导致 `ZMQ_CLIENT_STUB_SEND_LATENCY` 的含义从“实际发送耗时”退化为“排队耗时”，且与 `ZMQ_CLIENT_QUEUING_LATENCY` 严重重叠。
2. **P1 级命名冲突**：`SERVER_EXEC_NS` Tick 包含了服务端的排队等待时间，建议更名为 `SERVER_TOTAL_NS` 以避免误导性能调优。

---

### A. 功能与需求符合性

- **覆盖度**：实现了对 RPC 关键路径（Queuing, Stub Send, Server Wait, Exec, Reply, Network, E2E）的全面度量。
- **一致性**：通过 `RecordRpcLatencyMetrics` 确保了客户端能够感知到服务端内部的细分耗时。
- **缺口**：`SERVER_EXEC_NS` 的计算逻辑中包含了 `SERVER_QUEUE_WAIT`，这在分析“纯业务处理耗时”时会产生干扰。

### B. 性能与可扩展性

- **热路径优化**：
    - 在 `src/datasystem/common/rpc/zmq/zmq_stub_impl.h` 中，重构了 `RecordRpcLatencyMetrics`，将 5 次 Tick 查找合并为一次循环，有效降低了高并发 RPC 场景下的 CPU 开销。
    - 服务端 `RecordServerLatencyMetrics` 也采用了单次遍历。
- **内存开销**：`AsyncCallBack` 现在额外存储了一个 `MetaPb` 副本。对于包含大量 Tick 的复杂 RPC，这会增加内存驻留，但在观测性需求下是合理的权衡。

### C. 潜在缺陷（Bug / 竞态 / 资源）

#### P1 级问题

- **位置**：`src/datasystem/common/rpc/zmq/zmq_stub_impl.h:151` 与 `src/datasystem/common/rpc/zmq/zmq_stub_conn.cpp:124`
- **摘录**：
```cpp
// zmq_stub_impl.h
+        RecordTick(p.first, TICK_CLIENT_SEND);
+        RecordTick(p.first, TICK_CLIENT_STUB_SEND);

// zmq_stub_conn.cpp
-    RecordTick(meta, TICK_CLIENT_SEND);
```
- **问题**：`TICK_CLIENT_SEND` 原本在 `ZmqFrontend` 实际向 Socket 发送时记录，反映真实 I/O 开始。现移至 `AsyncWrite`（入队前），使其失去了原有的“网络发送起点”含义。
- **建议**：保留 `zmq_stub_conn.cpp` 中的 `TICK_CLIENT_SEND` 记录，或在 `AsyncWrite` 中使用新的 Tick 名称（如 `CLIENT_PRE_ENQUEUE`）。

- **位置**：`src/datasystem/common/rpc/zmq/zmq_constants.h:130`
- **摘录**：
```cpp
+    uint64_t serverExecNs = (serverExecEndTs > serverRecvTs) ? (serverExecEndTs - serverRecvTs) : 0;
+    execTick.set_tick_name("SERVER_EXEC_NS");
```
- **问题**：此处 `SERVER_EXEC_NS` 实际上是 `EXEC + QUEUE_WAIT`。在 `RecordRpcLatencyMetrics` 中，`NETWORK = E2E - SERVER_EXEC_NS`。如果 `EXEC_NS` 包含排队时间，则 `NETWORK` 指标将不包含服务端排队，这与通用定义（Network = E2E - ServerSideTotal）虽一致，但 `EXEC_NS` 命名极具误导性，容易让开发者认为业务逻辑很慢，而忽略了排队瓶颈。
- **建议**：更名为 `SERVER_TOTAL_NS`。

#### P2 级问题

- **位置**：`src/datasystem/common/rpc/zmq/zmq_stub_impl.h:224`
- **摘录**：
```cpp
+    for (int i = 0; i < clientMeta.ticks_size(); i++) {
+        TickPb *tick = rsp.first.mutable_ticks()->Add();
+        *tick = clientMeta.ticks(i);
+    }
```
- **问题**：手动循环合并 Protobuf 重复字段效率较低，且未处理重复 Tick。
- **建议**：改用 `rsp.first.mutable_ticks()->MergeFrom(clientMeta.ticks());`，并考虑是否需要过滤重复的 Tick 以节省 Meta 大小。

### D. 代码风格与主仓一致性

- **常量定义**：`zmq_constants.h` 中引入了 `kNsPerUs` 等常量，符合既有命名规范。
- **RAII/安全性**：`AsyncCallBack` 对 `MetaPb` 的持有采用了 `std::move`，符合资源管理规范。
- **头文件组织**：`RecordServerLatencyMetrics` 作为 `static inline` 定义在 `.h` 中，虽然增加了编译依赖，但有利于编译器优化热路径下的 Tick 解析逻辑。

### E. 具体优化与重构（可执行清单）

1. **[E-1] [P1] 维度：功能/性能**
    - 位置：`src/datasystem/common/rpc/zmq/zmq_stub_impl.h:91`
    - 摘录：`RecordLatencyMetric(metrics::KvMetricId::ZMQ_CLIENT_STUB_SEND_LATENCY, clientToStubTs - clientStubSendTs);`
    - 问题：`clientStubSendTs` 和 `clientEnqueueTs` 都在 `AsyncWrite` 连续记录，导致两个指标几乎完全一致。
    - 建议：修正 `ZMQ_CLIENT_STUB_SEND_LATENCY` 的计算公式，使其覆盖 Frontend 处理到真正 `zmq_send` 的时间，即 `TICK_CLIENT_SEND - TICK_CLIENT_TO_STUB`（需恢复 `zmq_stub_conn.cpp` 中的 Tick 记录）。

2. **[E-2] [P2] 维度：风格**
    - 位置：`src/datasystem/common/rpc/zmq/zmq_constants.h:97`
    - 建议：将 `RecordServerLatencyMetrics` 的逻辑进一步封装，避免在头文件中直接暴露过于复杂的逻辑，或者确保其在所有调用点都能被有效内联。
