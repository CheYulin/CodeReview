## Executive Summary

本 PR 旨在通过细化 RPC 关键路径的 Tick 打点，建立完整的 ZMQ RPC 队列流转监控体系。主要变更包括在客户端 Stub 和服务端 Service/Stream 中引入更多维度的延迟度量（如 `CLIENT_STUB_SEND`、`SERVER_QUEUE_WAIT` 等），并修正了 E2E 统计逻辑。

**风险轮廓：**
- **性能风险（高）**：在 RPC 响应的热路径中直接调用了使用 `LOG(ERROR)` 的调试函数 `DumpServerTicks` / `DumpClientTicks`，且未受开关控制。这将导致生产环境下严重的性能退化和日志灌满。
- **效率优化（中）**：客户端指标记录存在多次重复遍历 Protobuf 列表的操作。
- **可维护性（低）**：`zmq_constants.h` 引入了具体的指标记录逻辑和较重的头文件依赖，建议进一步解耦。

**合入结论：建议阻塞（修改后通过）**。必须移除或开关化热路径中的全量日志打印。

---

## A. 功能与需求符合性

- **监控指标完整性**：成功实现了从客户端入队到服务端发送回包的全链路 7 个核心指标记录，涵盖了 `ZMQ_CLIENT_QUEUING_LATENCY` 到 `ZMQ_RPC_NETWORK_LATENCY`。
- **Bug 修复**：在 `zmq_stub_impl.h` 中修正了 `e2eNs` 的计算方式，改用明确的 `CLIENT_RECV - CLIENT_ENQUEUE`，避免了旧逻辑中因 `tick[0]` 时间戳异常导致的统计错误。
- **覆盖范围**：变更不仅覆盖了常规 Service，还兼顾了 `zmq_server_stream_base.h` 中的流式接口。

---

## B. 性能与可扩展性

- **热路径开销**：`RecordServerLatencyMetrics` 被引入 `zmq_constants.h` 并被 `ZmqService` 和 `StreamBase` 调用。虽然该函数逻辑清晰，但在高频 RPC 场景下，频繁执行 `meta.mutable_ticks()->Add()` 会增加 Protobuf 序列化的负担。
- **日志瓶颈**：`DumpServerTicks` 等调试函数使用 `LOG(ERROR)` 级别。在分布式系统中，`LOG(ERROR)` 通常涉及刷盘或同步操作，会使 RPC 吞吐量下降几个数量级。

## C. 潜在缺陷（Bug / 竞态 / 资源）

### P1 级：热路径无条件日志打印
- 位置：`src/datasystem/common/rpc/zmq/zmq_service.cpp:1011` 与 `src/datasystem/common/rpc/zmq/zmq_stub_impl.h:224`
- 摘录：
```cpp
+    // Record SERVER_SEND tick and calculate server-side metrics
+    RecordTick(meta, TICK_SERVER_SEND);
+    DumpServerTicks(meta);
+    RecordServerLatencyMetrics(meta);
```
- 问题：`DumpServerTicks` 内部使用 `LOG(ERROR)` 循环打印所有 Tick。在正常请求流中，这不符合“Debug helper”的初衷，且严重影响性能。
- 建议：使用 `VLOG` 或增加配置开关控制，或者仅在采样/出错时打印。

### P2 级：多重遍历性能损耗
- 位置：`src/datasystem/common/rpc/zmq/zmq_stub_impl.h:74-95`
- 摘录：
```cpp
+    uint64_t clientEnqueueTs = FindTickTs(meta, TICK_CLIENT_ENQUEUE);
+    uint64_t clientRecvTs = FindTickTs(meta, TICK_CLIENT_RECV);
...
+    uint64_t clientToStubTs = FindTickTs(meta, TICK_CLIENT_TO_STUB);
+    uint64_t clientSendTs = FindTickTs(meta, TICK_CLIENT_SEND);
```
- 问题：`RecordRpcLatencyMetrics` 内部 5 次调用 `FindTickTs`，每次都会遍历 `meta.ticks()` 列表。当 Tick 数量较多时，该 O(M*N) 的操作是不必要的。
- 建议：仿照 `RecordServerLatencyMetrics` 的写法，单次遍历 Tick 列表并根据 `tick_name` 填充局部变量。

---

## D. 代码风格与主仓一致性

- **头文件依赖**：`zmq_constants.h` 传统上仅存放常量定义，现在引入了 `#include "datasystem/common/metrics/kv_metrics.h"` 并包含复杂的 `static inline` 函数。这增加了该基础头文件的耦合度。
- **单例模式使用**：符合项目中通过 `metrics::GetHistogram` 获取指标实例的既有风格。

## E. 具体优化与重构（可执行清单）

[E-1] [P1] 维度：性能
- 位置：`src/datasystem/common/rpc/zmq/zmq_service.cpp:1011`
- 摘录：`+    DumpServerTicks(meta);`
- 问题：无条件执行高开销日志打印。
- 建议：删除此行，或将其封装在 `if (FLAGS_ds_rpc_debug_latency)` 等开关下。

[E-2] [P2] 维度：性能/风格
- 位置：`src/datasystem/common/rpc/zmq/zmq_stub_impl.h:74`
- 摘录：多次调用 `FindTickTs`。
- 问题：冗余的 Protobuf 列表遍历。
- 建议：优化为单次循环遍历：
```cpp
+    uint64_t enq=0, recv=0, toStub=0, send=0;
+    for (const auto& t : meta.ticks()) {
+        if (t.name() == ...) enq = t.ts();
+        // ...
+    }
```

[E-3] [P2] 维度：功能/一致性
- 位置：`src/datasystem/common/rpc/zmq/zmq_constants.h:131`
- 摘录：`+    execTick.set_tick_name("SERVER_EXEC_NS");`
- 问题：测试用例注释提到 `SERVER_EXEC_NS` 实际上包含了 `QUEUE_WAIT` 时间（`EXEC_END - RECV`）。如果目的是度量“服务端总时长”，命名尚可；若为“执行时长”，则逻辑有偏。
- 建议：在代码注释中明确 `SERVER_EXEC_NS` 的统计口径，以防后续维护者误解。

**修改后通过**。
