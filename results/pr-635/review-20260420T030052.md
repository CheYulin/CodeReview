### Executive Summary
本 PR 旨在增强分布式对象存储系统的可观测性，通过在 Client、Worker 和 Master 全链路埋点，解决 **Issue #393 (SHM Leak)** 和 **Issue #399 (TTL Observability)** 描述的资源泄露定位难题。
- **风险轮廓**：低风险。变更主要为指标采集，不涉及核心业务逻辑修改。
- **核心价值**：引入了“泄露指纹”概念，通过 Counter 与 Gauge 的组合（如 `SHM_REF_ADD` vs `SHM_REF_REMOVE`）能精准识别是任务堆积还是引用计数归零异常。
- **优先处理项**：
  1. 确认大容量场景下 `metaTable_.size()` 的调用开销（Master 侧）。
  2. 核对热路径（`DecreaseReferenceCnt`）在高频跳过时的原子操作性能影响。
  3. 建议将 `METRIC_INC` 宏统一用于所有 Counter 类型，减少模板代码。

---

### A. 功能与需求符合性
- **Traceability 符合性**：
    - **Issue #393 (SHM Leak)**：通过 `WORKER_ALLOCATOR_ALLOC_BYTES_TOTAL` 与 `WORKER_SHM_REF_TABLE_BYTES` 的对比，可直观判断物理内存分配与逻辑引用计数的脱节点。符合预期。
    - **Issue #399 (TTL Observability)**：在 `ExpiredObjectManager` 中增加了 `MASTER_TTL_PENDING_SIZE` 和 `MASTER_TTL_FIRE_TOTAL` 等指标，覆盖了从插入、触发到执行结果（成功/失败/重试）的完整生命周期。符合预期。
- **覆盖完备性**：新增的单元测试（Phase 1-3）模拟了“Standby 切换导致的静默丢弃”和“TTL 链条阻塞”等典型故障指纹，具备极高的诊断参考价值。

### B. 性能与可扩展性
- **指标开销**：绝大部分 Counter 使用 `METRIC_INC/ADD`（原子操作），对数据面热路径影响极小。
- **Master 侧负载**：`ExpiredObjectManager::Run` 中对 `GetMetaTableSize()` 的调用频率被限制为 1 秒一次（`kMetaTableSizeTickMs`），避免了在大规模元数据场景下频繁扫描并发容器导致的 CPU 抖动。
- **Client 侧负载**：`StartMetricsThread` 在后台线程更新 Gauge，且使用了 `shutdownMux_` 的读锁，符合线程安全与性能要求。

### C. 潜在缺陷（Bug / 竞态 / 资源）
#### P2 级别
1. **位置**：`src/datasystem/master/object_cache/oc_metadata_manager.h:1104`
   - **摘录**：
     ```cpp
     +    size_t GetMetaTableSize() const
     +    {
     +        return metaTable_.size();
     +    }
     ```
   - **问题**：若 `metaTable_` 是基于 `tbb::concurrent_hash_map` 实现，其 `size()` 操作在部分 TBB 版本下并非 $O(1)$，而是线性扫描 bucket，在海量键（如千万级）下可能导致该背景线程瞬间 CPU 占用过高。
   - **建议**：建议 Master 侧维护一个 `std::atomic<size_t>` 计数器随增删更新，或确认当前环境中 TBB `size()` 的开销。

2. **位置**：`src/datasystem/client/object_cache/object_client_impl.cpp:1497`
   - **摘录**：
     ```cpp
     +        METRIC_INC(metrics::KvMetricId::CLIENT_DEC_REF_SKIPPED_TOTAL);
     ```
   - **问题**：在 `DecreaseReferenceCnt` 这个极高频调用的函数入口处增加原子自增。如果业务逻辑触发大量 early-return（如 shmId 为空或 pool 关闭），原子操作的 Cache Line 竞争可能在高并发下变得显著。
   - **建议**：考虑仅在非平凡跳过（如 `!needDecreaseWorkerRef`）时记录，或接受此开销。

### D. 代码风格与主仓一致性
- **一致性**：新增的枚举值 `KvMetricId` 和描述数组 `KV_METRIC_DESCS` 严格遵循了既有的指标管理模式。
- **RAII/生命周期**：在 `ObjectClientImpl::StartMetricsThread` 中使用了 `std::shared_ptr<ThreadPool> pool = asyncReleasePool_;` 并在锁外调用，正确处理了对象生命周期。
