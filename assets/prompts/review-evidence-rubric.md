## 输出结构（必须严格遵守）

先给一段 **Executive summary**（≤10 行）：PR 风险轮廓、是否建议阻塞合入、最需优先处理的 3 点。

然后按下列顺序输出；**无问题**的维度写「本 diff 中未见需报告项」并说明理由。

### A. 功能与需求符合性

- 逐条对照 **AR1（#234）** 及与 **#233 / #235** 的前瞻关系。
- 每条结论若有风险，按 **证据规则** 给出摘录 + 建议。

### B. 性能与可扩展性

- 仅基于 diff 可推断的路径：锁粒度、分配、序列化、RPC/元数据频率等。
- 若需 profiling：明确写出 **测什么指标、在哪个接口/路径**。

### C. 潜在缺陷（Bug / 竞态 / 资源）

- 按 **P0 → P3** 分组；同级内按文件名字母序。

### D. 代码风格与主仓一致性

- 对比 diff 内**同一文件**修改前后或**相邻未改行**的既有风格（若可见）。
- 风格问题也须带摘录；纯主观 nit 归入 P3。

### E. 具体优化与重构（可执行清单）

- 使用 **编号列表**；每一项格式固定为：

```text
[E-序号] [P0|P1|P2|P3] 维度：功能|性能|缺陷|风格
- 位置：<path>:<approx_line_in_diff>
- 摘录：（代码块，来自 diff）
- 问题：<一句话>
- 建议：<具体改法，可含伪代码或签名>
```

### F. 合入结论

- **通过 / 修改后通过 / 不通过** 三选一，并列出**阻塞项**编号（若有）。

## 摘录格式示例（结构示例，勿复制为真实问题）

````markdown
- 位置：`src/foo/bar.cpp`（diff 中 `+` 行附近）
- 摘录：
```cpp
+ if (!ptr) return;
+ DoWork(ptr->x);  // 若 ptr 在并发下失效？
```
- 问题：在持锁范围外使用 `ptr`，与 AR1 要求的崩溃一致性不一致。
- 建议：在 `Lock()` 内复制所需字段或使用 `shared_ptr` 延长生命周期（与 `baz.cpp` 中 `Handle` 模式对齐）。
````
